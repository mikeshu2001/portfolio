# Спецификация: логирование

Версия 1.0. Собрано 2026-04-21.

## Зачем

Портфолио — учебный пример для курса по веб-кодингу. Логирование здесь демонстрирует паттерны, которые студенты переиспользуют в своих реальных проектах: централизованный модуль с уровнями, структурированные события, защита от утечки PII, тестируемость.

На этом сайте логи идут **только в консоль браузера**. Сторонние сервисы (Sentry и т. п.) и свой сервер не подключаем.

## Что логируем

| Уровень | Событие                   | Где                              | Поля                                                    |
| ------- | ------------------------- | -------------------------------- | ------------------------------------------------------- |
| info    | `page:loaded`             | `js/page-events.js`              | `path`, `referrer`, `userAgent`, `viewport`             |
| debug   | `form:counter:updated`    | `js/contact-form.js` на `input`  | `length`, `overflow`                                    |
| info    | `form:submit:attempt`     | `js/contact-form.js` на submit   | `messageLength`                                         |
| warn    | `form:validation:failed`  | `js/contact-form.js` при отказе  | `reason` (`too_long` / `invalid_fields`), `length`      |
| info    | `form:submit:success`     | `js/contact-form.js` после 200   | `durationMs`, `status`                                  |
| error   | `form:submit:error`       | `js/contact-form.js` в catch     | `errorMessage`, `durationMs`                            |
| error   | `uncaught:error`          | `js/page-events.js`              | `errorMessage`, `filename`, `lineno`, `colno`           |
| error   | `uncaught:rejection`      | `js/page-events.js`              | `reason`                                                |

Имена событий — через двоеточие (`namespace:object:action`). Это позволяет позже грепать логи или группировать по категории.

## Чего НЕ логируем

**Персональные данные.** Никогда. Значения полей формы — имя, email, telegram, сообщение — в логи не попадают.

Реализация: функция `sanitize(data)` в модуле. Если у поля в объекте имя из списка `['name', 'email', 'telegram', 'message']` (case-insensitive) — значение заменяется на строку `'[redacted]'`. Работает рекурсивно для вложенных объектов. Благодаря этой защите даже случайный `logger.info('form:submit', { email: user.email })` не утечёт email в консоль.

## Модуль `js/logger.js`

Четыре уровня: `debug`, `info`, `warn`, `error`. Вывод — структурированный:

    [2026-04-21T18:32:45.123Z] INFO  form:submit:success  { durationMs: 1234 }

С цветовой подсветкой уровня (`%c`-форматирование), чтобы в DevTools было читаемо.

### API

- `window.logger.debug(event, data?)` и аналогично `.info`, `.warn`, `.error` — готовый инстанс с уровнем `debug`. Подключается на странице автоматически.
- `createLogger(minLevel, sink)` — фабрика для тестов и нестандартных сценариев. `sink` по умолчанию пишет в консоль; в тестах подменяется на функцию-сборщик, без монкипатчинга `console`.
- `sanitize(data)` и `formatEvent(level, event, data)` — экспортируются отдельно для тестов.

## Уровни и фильтрация

По умолчанию сайт стартует с `debug` — поток событий видно весь, включая шум вроде обновления счётчика. В настоящем проде ставят `info` или `warn`; это показывается в документации и комментариях, чтобы студенты увидели, где переключается.

## TDD-план

Тесты в `tests/logger.test.js` на чистые функции:

- `sanitize` (5 тестов): замена PII, case-insensitive, вложенность, неизменность входа, не-объекты пропускаются как есть.
- `formatEvent` (3 теста): корректная структура с `time`/`level`/`event`/`data`, санитизация данных, дефолт для отсутствующего `data`.
- `createLogger` (4 теста): наличие методов `debug/info/warn/error`, sink получает payload, level-фильтрация, санитизация в полном pipeline.

Красная фаза: тесты падают, потому что модуля нет → пишу модуль → зелёная фаза.

## Что оставлено на потом

- **Отправка логов на сервер.** Когда появится (Sentry, свой бэкенд) — меняется только `defaultSink` в модуле, все call sites остаются без правок.
- **Сохранение в localStorage** на случай, если консоль закрыта. Не делаем: студент, смотрящий пример, всё равно откроет DevTools.
- **Панель логов на странице** (visible log viewer). Возможное расширение как отдельный урок курса.
