# Спецификация: ограничение длины сообщения

Версия 1.0. Собрано в диалоге 2026-04-21.

## Обзор

Сообщение в форме контактов не должно быть длиннее 1000 символов. Превышение блокирует отправку и наглядно показывает это пользователю.

## Что видит посетитель

1. Под полем «Сообщение» появляется счётчик вида `0 / 1000`. Обновляется при каждом нажатии клавиши / вставке текста.
2. При длине от 1 до 1000 включительно — счётчик серый (цвет второстепенного текста).
3. При длине ≥ 1001 — счётчик становится красным, вместо цифр текст «N / 1000 — слишком длинно», кнопка «Отправить» блокируется (не нажимается).
4. Если пользователь всё-таки отправит длинный текст (например, вставил поверх уже разблокированной формы) — на обработчике submit появляется красный блок над формой: «Сообщение слишком длинное, максимум 1000 символов». Запрос к Formspree не уходит.

## Точная логика

- Считается `message.value.length` — длина строки в JavaScript. Для русского и латиницы совпадает с видимыми символами. Сложные эмодзи (например, 🧑‍🎤 — семья, флаги) могут считаться за несколько — это приемлемо для MVP.
- Верхняя граница: ровно 1000 символов — валидно, 1001 — уже ошибка.
- Нижняя граница: не проверяется (остаётся только `required` в HTML, запрет пустого поля).

## Как устроено технически

- **Чистая функция** `validateMessage(text)` в новом файле `js/validate-message.js`. Возвращает `{ ok: true }` или `{ ok: false, error: "Сообщение слишком длинное, максимум 1000 символов" }`. Не трогает DOM, не знает про форму — только строка на вход.
- **Тесты** в `tests/validate-message.test.js`. Встроенный в Node тест-раннер (`node --test`), без внешних зависимостей. Запуск: `npm test`. Проверяются граничные случаи: пустая строка, 999, 1000, 1001, длинный юникод.
- **`package.json`** добавляется с одним скриптом `test`. Это не «сборка» — только чтобы команда `npm test` знала, что запускать.
- **Интеграция в форму** — обработчик в `js/contact-form.js` вызывает `validateMessage`:
  - на событии `input` в textarea → обновляет счётчик и состояние кнопки;
  - на `submit` → повторная проверка перед fetch, защита от гонки.

## Тексты

- Счётчик в норме: `15 / 1000`
- Счётчик при превышении: `1023 / 1000 — слишком длинно`
- Блок ошибки (тот же `#form-error`): «Сообщение слишком длинное, максимум 1000 символов.»

## Что оставлено на потом

- Минимум длины сообщения — не добавляем, вне задачи.
- Счётчики на другие поля (имя, email, telegram) — не нужны, их форматы уже ограничены типами.
- Подсветка самого textarea (не только счётчика) — если UX счётчика окажется недостаточным, можно добавить.
